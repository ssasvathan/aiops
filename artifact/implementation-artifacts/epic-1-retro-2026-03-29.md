## Challenges and Growth Areas

### 1. Docker Environment Setup Friction (Story 1.1)

Bob (Scrum Master): "In Story 1.1, Docker Desktop wasn't running. The developer had to start `docker-desktop.service` to get testcontainers working. This blocked the regression gate on first run."

Amelia (Dev): "And unrelated failures in Story 3.1 ATDD fixtures and the Prometheus integration test needed fixing to unblock the full suite. Scope bled slightly."

**Root Cause:** Docker pre-check was not standardized in the development onboarding flow.

### 2. Scoring Payload Validation Took Multiple Code Review Cycles

Murat (TEA): "Non-finite scores, boolean coercion, whitespace in reason codes — these payload hardening issues were caught across Story 1.2's code review in rounds. We should have a scoring payload validation checklist."

Bob (Scrum Master): "Seven findings in Story 1.2, nine in Story 1.3. Pattern suggests implementation outpaces defensive input validation habits."

**Root Cause:** No pre-flight checklist for scoring boundary/edge cases and strict-parse patterns.

### 3. Story 1.3 Scope Ambiguity Risk

Alice (Product Owner): "Story 1.3 covered FR11/FR18/FR19 but the primary codebase wiring was already done by 1.1 and 1.2. The story was essentially verification and audit-outcome quality — minimal new code."

Amelia (Dev): "It wasn't ambiguous in the story doc, but the risk of over-building was real. Keeping it to test extension + behavioral verification was the right call."

---

## Key Insights and Learnings

1. **Strict payload parsing from day one** — Defensive validation (non-finite rejection, bool-not-float, reason-code normalization) should be written in the initial implementation pass, not deferred to code review.

2. **Docker pre-check is a dev setup step, not an optional check** — Add as the first task in any story requiring testcontainers.

3. **ATDD red-phase + frozen contracts = safe incremental delivery** — The combination of failing tests first and not touching contracts created a safe delivery corridor across three stories.

4. **Code review is the safety net but should not be the only net** — HIGH findings in back-to-back stories indicates a pattern. Pre-flight checklist items for scoring logic and metadata validation would reduce review load.

5. **Story 3.1 ATDD fixtures were latent broken** — Discovered and fixed during Story 1.1. Cross-epic dependency health should be verified as part of story kickoff.
