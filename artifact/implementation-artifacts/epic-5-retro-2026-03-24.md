---

### Key Insights and Learnings

1. **Policy architecture has a repeatable composition pattern**: define contract → create YAML → load at startup → wire into pipeline → stamp in casefile. Stories 5.1 and 5.3 followed this pattern explicitly. Future epics adding new policy dimensions should use this as a template.
2. **Pre-story source analysis prevents duplicate implementation**: Story 5.4's dev agent performed a comprehensive source scan and identified pre-existing implementations early. This should be a standard step in the create-story workflow.
3. **Collect-all-failures before raising is correct validator ergonomics**: The L3 finding in 5.2 (raising on first failure rather than collecting all failures) is a recurring anti-pattern in validators. The correct pattern (iterate, collect, raise once) should be added to the architecture implementation-patterns guide.
4. **Review finding symmetry check**: L4 in 5.2 (testing non-prod bypass for PD but not Slack/SN) is a test coverage pattern failure. Whenever a validator iterates over multiple integration types, test coverage must be symmetric across all types. This should be an explicit checklist item for validators.
5. **Acceptance-note artifact requires tooling, not discipline**: Two consecutive epics have missed this commitment. The reusable template exists (`artifact/epic-acceptance-note-template.md`), but there is no workflow gate enforcing its creation. Without a workflow hook, this will continue to be skipped.

## 2) Deep Story Pattern Analysis

### Common Struggles

- Code review returned changes-requested in all 4 stories (100%) — this is the consistent overhead per story.
- Test assertion completeness (missing symmetric coverage, missing defensive assertions in "succeeds" tests) was flagged in 3 of 4 stories (5.1, 5.2, 5.4).
- Naming discipline (unconventional local variable names, misleading comments) was flagged in 5.1 and 5.2 — both resolved quickly.

### Recurring Review Feedback

- **Missing symmetric test coverage**: If a validator covers 3 integration types, tests must cover all 3 for both the positive and negative path. This pattern was missed in 5.2 and reflects an implicit assumption that "similar" cases can be inferred rather than tested.
- **Missing defensive assertions in "succeeds" tests**: Several tests verified the success case without asserting the resolved environment value (e.g., `APP_ENV.value == "prod"`). This is the M2 pattern across 5.1 and 5.2.
- **Misleading structural comments in `__init__.py`**: 5.1 had a split contracts header that was alphabetically inconsistent. Small, but easy to prevent with a lint-level check.

### Breakthrough Moments

- Story 5.3 completing the anomaly detection policy wiring: operators can now edit `anomaly-detection-policy-v1.yaml` and change detection thresholds without touching code. This was the capstone of a design pattern started in Epic 1 that progressively externalized constants.
- Story 5.4 discovering that most of the implementation was already in source: this is a strong signal that the team is building ahead and that ATDD/design-first discipline is working.

### Velocity Patterns

- Stories progressed sequentially and cleanly: 5.1 (contract + startup) → 5.2 (security validators) → 5.3 (policy wiring + version stamping) → 5.4 (health + observability).
- Each story closed with a single review round. Review overhead is stable but not improving further.
- Test count growth (+7, +8, +8, +9) is consistent — each story adds targeted coverage without test bloat.

### Testing and Quality Insights

- Zero-skipped gate held. This is now a team norm.
- OTLP metric assertion coverage was present in every story that touched metrics — Epic 4 retro action item fully absorbed.
- Integration test quality concerns (the primary Epic 4 finding) were absent in Epic 5 — the skip/fail discipline and metric assertion requirement were internalized.
- The remaining review friction is in unit test design: symmetric coverage, defensive assertions, consistent naming.
