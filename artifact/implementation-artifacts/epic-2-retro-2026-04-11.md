### What Went Well

**The story-to-story knowledge transfer was exceptional.**
Each story carried a "Previous Story Learnings" section in dev notes that directly reflected the prior story's review findings. Story 2-2 explicitly listed 5 learnings from 2-1 and applied them from the start. Story 2-3 listed 5 learnings from both 2-1 and 2-2. As a result, the per-story review patch count dropped: 7 patches in 2-1 (5 fixed + 2 deferred), 4 in 2-2 (3 + 1 deferred), 3 in 2-3 (3 + 0 deferred). Clear learning trajectory across the epic.

**WCAG AA compliance was baked in from story 2-1 review and carried forward.**
The hero banner initially showed raw 0/1/2 numerics without text labels — a real WCAG gap caught in review. The fix (value mappings 0→HEALTHY, 1→DEGRADED, 2→UNAVAILABLE) was documented in the 2-1 learnings and applied proactively in stories 2-2 and 2-3. The color palette is muted and professional throughout, and forbidden Grafana defaults are tested with case-insensitive checks. A complete WCAG AA posture was established by story 2-2.

**Stat panel as "heatmap" was the right architectural call.**
Story 2-2 chose `type: "stat"` with `reduceOptions.values: true` over `type: "heatmap"` for the topic tiles. This was a deliberate, well-documented decision (Grafana's `heatmap` panel type is for time-bucketed density, not per-label health tiles). The decision was correct, the rationale was captured, and the implementation was clean on the first pass.

**Dashboard version test evolution prevents story-to-story regressions.**
Story 2-1 added `test_dashboard_version_is_2`. Story 2-2 updated the 2-1 test to `>= 2` and added `test_dashboard_version_is_3`. Story 2-3 updated the 2-2 test to `>= 3` and added `>= 4`. This pattern evolution — from strict equality to `>=` — was discovered through practice and is now an established convention. It prevents a future story from having to go back and patch all prior version tests.

**Config-validation test pattern scaled perfectly.**
The entire test suite for both epics uses zero docker-compose. All tests parse static JSON or YAML from disk. 88 dashboard + infra integration tests run in milliseconds with no external service dependencies. This was set up in Epic 1 and held perfectly across Epic 2.

**Zero debug log entries across all 3 stories.**
All three 2-x stories reported "implementation was straightforward / no blockers / no debug issues." This is a testament to the quality of the dev notes, the established patterns, and the architecture's clarity. The Epic 1 investment in detailed panel JSON patterns paid off directly in Epic 2's clean execution.

### Challenges and Growth Areas

**Transparent background handling was subtler than expected.**
Story 2-1 set `options.background: "transparent"` as an explicit field. Story 2-2's dev notes corrected this: in Grafana 12.4.2, `colorMode: "background"` on stat panels handles background color behavior; the dark dashboard background (`#181b1f`) serves as the visual separator. Story 2-3 still missed `"transparent": true` on the timeseries panel (id=5) — caught as an H1 (High severity) finding in review. The `transparent: true` behavior differs between stat panels and timeseries panels, and this distinction wasn't fully codified in the dev notes until 2-3's review. Three stories, three slightly different transparency approaches.

**Font size requirement for heatmap tiles was missed on first pass.**
Story 2-2's review caught a missing tile font size configuration (AC4/UX-DR2 requires 14px+ for heatmap tile labels). This was an H1 finding that required a patch adding `options.text: {titleSize: 14, valueSize: 16}` and a corresponding test. The UX-DR2 requirement was in the acceptance criteria but the dev agent did not apply it. A clear case where the checklist should have caught it.

**The `or vector(0)` fallback with `values: true` creates a labelless zero-tile.**
This was deferred in 2-2 (D1 finding). The `or vector(0)` pattern that prevents "No data" states in stat panels works differently in multi-series mode (`values: true`): if no topic data exists, it renders a single unlabelled green tile. This is an edge case that only manifests in pure dev environments with no Kafka data, but it's a known cosmetic issue. The deferred finding is documented and understood.

**gridPos.x == 0 invariant was not consistently tested.**
Story 2-3's review added `x == 0` assertions to fold separator and baseline overlay position tests (L1 finding). The full-width panel invariant (`x: 0, w: 24`) was explicitly stated in dev notes for every story but only tested for `w` and `y`, not `x`. Three stories went by before this gap was caught.

**Version test strict equality pattern persisted until caught in review.**
Story 2-3's initial implementation used `== 4` (strict equality) for `test_dashboard_version_is_4`. Review M1 flagged this as breaking future stories and changed it to `>= 4`. This was the same pattern that was established in 2-2, but the dev agent implemented the 2-3 version test with the old strict pattern. The `>=` convention is now in the learnings for both 2-2 and 2-3 but should be codified as a rule in `project-context.md`.

---

### Recurring Review Feedback Patterns

Across all three stories, the following themes appeared in code reviews:

1. **Transparent backgrounds (H1 in 2-3):** The `transparent: true` property is required on all panels but behaves differently by panel type. Stat panels use `colorMode` + dark background; timeseries panels need explicit `"transparent": true`. This must be in the panel JSON checklist.

2. **WCAG value mappings (H1 in 2-1, applied proactively in 2-2, 2-3):** Color-coded panels must have text value mappings. Caught once, applied in all subsequent stories. Now established.

3. **Font size configuration (H1 in 2-2):** Tile/panel font sizes must meet UX-DR2 minimums. Must be explicitly tested with a `text.titleSize` / `text.valueSize` assertion.

4. **Version test `>=` not `==` (M1 in 2-3):** Dashboard version tests must use `>=` to remain valid as subsequent stories increment the version. Should be enforced as a linting convention.

5. **Stale docstrings and comments (M1 in 2-2, from 2-1 pattern):** TDD red-phase comments or intent comments in test classes must be removed after passing. This appeared in 2-1 review and recurred in 2-2.

6. **Missing position invariant tests (L1 in 2-3):** `gridPos.x == 0` was not tested until story 2-3 review. Full-width invariant tests should cover all three of `x`, `y`, `w` (and `h`).

---

### Key Insights and Learnings

1. **The `>=` version test convention is a forcing function for story independence.** When each story's test uses `>= N` instead of `== N`, the test remains valid as the dashboard grows. This should be codified in `project-context.md` as a project rule, not a per-story convention.

2. **Transparent backgrounds need panel-type-aware documentation.** The behavior of `transparent: true` vs. `colorMode: "background"` is not intuitive across panel types. The dev notes pattern for each panel type (stat, timeseries, text) should explicitly state the correct transparency approach.

3. **Story-to-story dev notes carry the highest ROI of any Epic 2 artifact.** The review patch count declining from 7 → 4 → 3 across the three stories is directly attributable to the "Previous Story Learnings" section being populated with real review findings and applied proactively. This pattern should be mandatory for all future epics.

4. **Config-validation tests with no live stack are fully viable for dashboard verification.** The entire Epic 2 visual narrative can be validated by parsing JSON. No docker-compose needed. This keeps CI fast and removes infrastructure flakiness from the test suite.

5. **The MVP baseline band (visual-only, synthetic +30% envelope) is a conscious technical debt item.** The architecture explicitly defers real seasonal baselines (Redis, Phase 3) to later work. The current implementation uses `rate * 1.3 + 0.1` as the upper bound. This is documented and understood. It must be surfaced in Epic 3/4 planning when baseline accuracy becomes business-critical.

6. **The hero banner proxy query (`sum(aiops_findings_total)`) is a placeholder, not a real health signal.** It was acknowledged in both the panel description and dev notes from story 2-1 onwards. A dedicated recording rule or health gauge is deferred. This will surface as a confusing "green even under failures" behavior in live demos if not addressed before Epic 5 (demo-ready validation).
