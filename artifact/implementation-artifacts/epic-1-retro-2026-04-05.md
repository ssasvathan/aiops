## Challenges and Areas to Improve

### 1. Circular Import Constraint Required Deferred Import in `bulk_recompute()`

Amelia (Dev): "Story 1.4 required a deferred `import` of `_build_best_effort_scope` inside `bulk_recompute()` due to a circular import between `baseline/client.py` and `pipeline/baseline_backfill.py`. The code review caught the missing explanation comment."

Winston (Architect): "This is a structural tension from `SeasonalBaselineClient` needing a helper that lives in the pipeline layer. The import inversion exists because we didn't surface this in the architecture. Story 2.x will inherit this constraint — it should be documented."

**Pattern observed once, resolved:** Deferred import documented with comment. Should be reconsidered in Epic 2 architecture prep.

### 3. `scope_count` Logging Bug Found in Review (Story 1.3)

Amelia (Dev): "In Story 1.3, the `scope_count` field in the backfill log event was incorrectly counting (scope, metric_key) pairs rather than unique scopes. The code review fixed this. It was a Medium finding."

Quinn (QA): "The test didn't catch it because we were testing that the event was emitted, not verifying the exact semantics of scope_count."

**Lesson:** Log event field semantics (unique scopes vs. scope-metric pairs) should be explicitly specified in acceptance criteria for Epic 2's structured log events.

---

### 4. `noqa` Comments for Inactive Ruff Rules Were Added Twice

Amelia (Dev): "Stories 1.2 and 1.3 both added `noqa` comments for `ARG002` and `BLE001` — rules that are NOT in the active ruff select set `E,F,I,N,W`. Both were caught in review."

Quinn (QA): "This suggests the dev agent doesn't have a reliable way to check which ruff rules are active before adding noqa comments."

**Lesson carried forward:** Add a note to story Dev Notes for Epic 2: 'Verify rule is in active ruff select set `E,F,I,N,W` before adding noqa.' This was already in Story 1.3's learnings — it just needs to be checked against earlier behavior.

---

### 5. Integration Tests Require Docker/Testcontainers — Not in Unit CI Pass

Quinn (QA): "The Story 1.3 integration tests in `tests/integration/test_baseline_deviation.py` require Docker. The full unit regression suite (`pytest tests/unit/`) passes at 1,225 tests with 0 skips, but the integration test suite requires a separate Docker environment."

Amelia (Dev): "This is consistent with how existing integration tests are structured in the project — it's not a regression. But it means there's a gap: the baseline seeding path through real Redis isn't exercised in the standard CI pass."

**Action item for Epic 2:** Ensure the CI pipeline guidance explicitly documents when integration tests are expected to run. This is not a blocker for Epic 2 but should be tracked.

---

### 6. `_PrometheusFailureTracker` Pattern Was Novel — Reconciled NFR-R4 with AC5

Amelia (Dev): "Story 1.4 had an interesting design tension: `bulk_recompute()` must NOT raise exceptions on per-metric failures (NFR-R4 — no partial writes), but `_run_baseline_recompute()` must detect failures and emit the failed log event. The solution was `_PrometheusFailureTracker` wrapping the Prometheus client."

Winston (Architect): "That's a creative solution. It preserves the clean separation: `bulk_recompute()` accumulates data silently, `_run_baseline_recompute()` uses the tracker to check if any failures occurred post-execution."

**Lesson:** When NFRs create conflicting forces (fail-safe vs. observability), wrapper patterns are the right abstraction layer. Document this for Epic 2's error handling patterns.

## Key Insights and Lessons Learned

### L1: Canonical Implementation Sketches in Dev Notes Eliminate Ambiguity

When the architecture pre-specifies exact function signatures, key patterns, and anti-patterns in the story's Dev Notes, implementation is deterministic. The `time_to_bucket()` canonical sketch, the `SeasonalBaselineClient` implementation sketch, and the `bulk_recompute()` two-phase sketch all resulted in first-pass implementations that required only quality polish, not architectural rework.

**Apply to Epic 2:** Story Dev Notes for `compute_modified_z_score()`, the MAD deviation stage function signature, and the `AnomalyFinding` extension should include canonical sketches.

---

### L2: Code Review Catches the Bug Class That Tests Can't See

Story 1.1's naive datetime guard, Story 1.2's tautological test assertion, Story 1.3's `scope_count` semantics error, and Story 1.4's `exc_info` with no active exception context — none of these were caught by the test suite. All were found by the AI code reviewer.

**Apply to Epic 2:** Code review remains mandatory for every story. The review effectiveness has been high (0 re-opened findings, all fixed same-day).

---

### L3: The "No Hardcoded Numbers" Rule (P2) Is Worth Enforcing Absolutely

Zero magic numbers escaped into the codebase. `MAD_CONSISTENCY_CONSTANT`, `MAD_THRESHOLD`, `MIN_BUCKET_SAMPLES`, `MAX_BUCKET_VALUES` — all imported from `constants.py` throughout. This makes Epic 2 stories that build on these constants (2-1, 2-3) cleanly testable with a single import change.

**Apply to Epic 2:** Continue P2 enforcement. Epic 2 may need additional constants for MAD computation edge cases (zero-MAD handling, precision thresholds) — add to `constants.py` immediately if needed.

---

### L4: Three-Layer Backfill Architecture Is Now Stable

The backfill now has three layers — Layer A (Redis peak cache), Layer B (in-memory peak history deque), Layer C (seasonal baseline buckets). Each layer is independently documented, testable, and wired. This is the foundation Epic 2 reads from.

**Apply to Epic 2:** The baseline deviation stage (Story 2.3) reads from Layer C. The `SeasonalBaselineClient.read_buckets_batch()` method is the right read path. No backfill changes needed in Epic 2.

---

### L5: Structured Log Event Names Are a Contract

The P6 rule (log events use `baseline_deviation_` prefix) was enforced across all stories. `baseline_deviation_backfill_seeded`, `baseline_deviation_recompute_started`, `baseline_deviation_recompute_completed`, `baseline_deviation_recompute_failed` are locked names. Consumers depend on them.

**Apply to Epic 2:** The Epic 2 stories will add `baseline_deviation_stage_started`, `baseline_deviation_finding_emitted`, `baseline_deviation_suppressed_*` events. These are pre-defined in P6 and must be implemented exactly.

---

### L6: Two-Phase In-Memory Compute + Bulk Write Pattern Is Sound

Story 1.4's `bulk_recompute()` proved the NFR-R4 pattern: compute everything in memory, write atomically via pipeline, never touch Redis during compute phase. This pattern should be the template for any future bulk data operation in this codebase.
