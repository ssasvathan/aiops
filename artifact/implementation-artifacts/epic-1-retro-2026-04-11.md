**Code review caught real correctness bugs, not just style issues.**
The AG6 outcome logic bug in story 1-2 (always emitting `"skip"` because `on_pass`/`on_fail` write to `postmortem_reason_codes` not `gate_reason_codes`) would have silently produced incorrect gating metrics in production. **Exception isolation pattern was adopted consistently.**
By story 1-3, the pattern of wrapping each OTLP metric emit call in its own `try-except` with a structured warning log was referenced from prior stories and applied correctly — even the review found it was consistently done. This is a direct result of story 1-2's code review teaching the pattern.

**Ruff exclusion scope was addressed proactively.**
Pre-existing ruff violations in BMAD framework directories (`.agents`, `_bmad`, `.claude`, `.cursor`) were causing noise. Adding targeted `per-file-ignores` exclusions was a low-effort, high-value quality-of-life fix that keeps the linting signal clean going forward.

### Challenges and Growth Areas

**AG6 outcome logic required non-trivial understanding of the gating internals.**
The initial implementation of `_emit_gating_evaluation_metrics()` in story 1-2 used the wrong field (`gate_reason_codes` instead of `postmortem_reason_codes`) to determine AG6 outcomes. This wasn't a careless mistake — it required understanding the difference between AG0–AG5 and AG6's branching logic. The fix required knowing about `postmortem_required`, `ag6_eligible` derivation from `gate_input.env` and `gate_input.criticality_tier`, and the AG0 firing status. The gating stage has significant hidden complexity that will trip up future implementors.

**Exception isolation patterns were not fully applied on first pass.**
Both story 1-2 (`record_finding` inside broad try-except) and story 1-3 (loop-level try-except in evidence.py) required code review to catch isolation boundary errors. The pattern is now established but needed two rounds of correction to reach the correct granularity (per-emit vs. per-loop vs. per-stage). The right isolation level is per-emit call, not per-loop and not per-stage.

**`_gate_reason_prefixes` dict rebuilt on every call — caught late.**
The dict was being rebuilt 7 times per gate_input on every evaluation loop. This was a performance issue that only came up in code review, not in unit tests (which can't measure call-frequency overhead). Module-level constants are the right pattern for any dict that doesn't change between calls.

**Evidence gauge cardinality and lifecycle required careful design.**
The evidence status gauge uses delta accounting with a `_current_evidence_status` dict keyed by `(scope, metric_key, topic)` — this is non-trivial state. The dev notes document two design options before settling on the simpler composite-key approach. The deferred finding about unbounded dict growth with topic churn is a known design gap that will need attention at scale.

**No ATDD red-phase tests were pre-written for story 1-3.**
Stories 1-1 and 1-2 both had pre-existing ATDD red-phase test files. Story 1-3 did not. While 15 tests were ultimately written and all passed on first run, the lack of upfront failing tests meant the dev agent had to self-define acceptance signals rather than drive against pre-validated failing tests. The ATDD-first discipline slipped for one story.

---

### Recurring Review Feedback Patterns

Across all three stories, the following patterns appeared in code reviews:

1. **Exception isolation granularity** (1-2 and 1-3): OTLP metric emission calls must each have their own `try-except`. Over-broad exception scopes were flagged twice.
2. **Module-level constants vs. per-call construction** (1-2): Any dict, set, or structure that is invariant between calls should be defined at module scope — not rebuilt inside functions or loops.
3. **Stale comments in test files** (1-1): The ATDD red-phase test file retained "RED PHASE: ... does not exist yet" comments after implementation. Cleanup of intent comments after the target passes should be standard practice.
4. **Missing file in File List** (1-1): The ATDD test file was not included in the story's File List. File List should be checked against actual `git diff --stat` output before marking review complete.

---

### Key Insights and Learnings

1. **Gating stage has hidden complexity that requires explicit documentation.** AG4/AG5/AG6 use different output fields than AG0–AG3. Any future work touching `evaluate_rulebook_gates` should start with a read of the gate branching logic, not just the function signature.

2. **Exception isolation level matters for observability correctness.** Over-broad try-except blocks in metric emission code don't just hide errors — they silently suppress all subsequent metrics in the same scope/loop. Per-emit isolation is the established project pattern.

3. **ATDD red-phase tests are a forcing function for precision.** When they were present (stories 1-1, 1-2), they defined exact expected behaviors before implementation. When absent (story 1-3), the dev had to self-define and the review process had to compensate. The pattern is worth maintaining.

4. **Delta accounting for gauges requires tracking previous state, not current state.** The evidence gauge lifecycle design (composite key by `(scope, metric_key, topic)` → current status) works, but it grows unbounded. The bounded alternative (pruning on topic churn) is deferred technical debt.

5. **Code review is producing real bug catches, not just style feedback.** Three of the six patches applied were correctness issues (AG6 logic, exception isolation, loop-level abort). The code review skill is delivering value and should be maintained as a mandatory step for each story.

6. **The two-track architecture (Infrastructure + Instrumentation) worked cleanly.** Track A (1-1 infrastructure) and Track B (1-2, 1-3 instrumentation) converged without friction. The explicit "Previous Story Intelligence" sections in dev notes carried knowledge forward within the epic.
