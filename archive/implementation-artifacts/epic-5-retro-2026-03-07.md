## Deep Story Analysis (Stories 5.1 to 5.9)

### What Went Well

1. Gate engine foundation (Story 5.1) was the multiplier.
   - AG0-AG6 sequential evaluation, monotonic reduction, and policy-order validation composed cleanly across 6 subsequent gating stories without regressions.

2. Zero-skip regression discipline held throughout the epic.
   - Every story ran the full suite with zero skipped tests before marking done.

3. Adversarial code review caught every HIGH-severity defect.
   - HIGH findings were resolved same day in every case (5.1, 5.5, 5.7, 5.9).

4. `action_fingerprint` as unified identity token.
   - Redis dedupe key = PD dedup_key = audit log field. One stable identity threading through the full action lifecycle.

5. End-to-end integration test (Story 5.9) validated the full pipeline as a coherent system.
   - Kafka, Postgres, Redis, MinIO all wired together. Invariants proven, not just assumed.

6. Denylist enforcement honored at every outbound boundary.
   - Story 5.8 applied `apply_denylist(...)` at the Slack notification boundary, consistent with architecture decision 2B.

### Challenges and Friction Points

1. Dead model anti-pattern appeared in 3/4 integration stories.
   - Story 5.5: `_REQUIRED_AG5_ACTION_KEYS` defined but unused.
   - Story 5.7: `PageTriggerPayload` defined but `_send_live` built raw dicts bypassing it.
   - Story 5.8: `NotificationEvent` defined but not instantiated in any production path until review.
   - Root cause: contract models defined early in story prep, serialization path implemented later — by then `_send_live` was already using raw dicts and no test forced the connection.

2. LIVE mode HTTP coverage systematically under-prioritized.
   - Stories 5.5 and 5.7 both shipped initial implementations with untested LIVE HTTP paths.
   - Root cause: LIVE mode coverage was listed as the last task, making it the first thing dropped under time pressure.

3. Policy/contract mismatches discovered mid-implementation.
   - Story 5.1: `rulebook-v1.yaml` used `stage` key, Settings used `uat` — required compatibility shim.
   - Story 5.5: `redis-ttl-policy-v1` had a single `dedupe_seconds` per env; FR33 required per-action TTLs — required policy model migration.
   - Root cause: policy artifacts created during architecture phase not stress-tested against FR semantics until a developer wired them.

4. Process documentation commitments not addressed (carried from Epic 4).
   - Local-development integration prerequisites guide: not completed.
   - Carry-forward accountability at epic kickoff: not completed.
   - Pattern: epic-level commitments without story-level owners consistently go unaddressed.

5. Integration test isolation gap caught in review.
   - Story 5.9 initially used `auto.offset.reset=earliest` with unique consumer groups in module-scoped Kafka, causing stale-message cross-test contamination. Caught in adversarial review. Fixed by filtering messages by `case_id`.

### Pattern Synthesis

**Recurring review feedback themes:**
- Dead code/model not exercised in production serialization path (3 stories)
- Missing audit log fields — mode, action, case_id (2 consecutive stories)
- Integration mode LIVE coverage gaps (2 stories)
- Policy/contract mismatches requiring mid-story corrections (2 stories)

**Breakthrough moments:**
- Story 5.1: Established the Stage 6 gate engine with strict policy-order validation and monotonic action reduction — foundation for all gating stories.
- Story 5.5: Atomic SET NX EX for race-safe dedupe + per-action TTL design + DegradedModeEvent signaling.
- Story 5.7: Insight connecting `action_fingerprint` → Redis dedup key → PD `dedup_key` — elegant traceability chain across the full pipeline.
- Story 5.9: End-to-end wiring proved the full hot-path is coherent — not just unit-tested silos.

**Velocity pattern:**
- Early stories (5.1-5.4): gate-engine focused, compact change sets, fast cycles
- Middle stories (5.5-5.8): integration-heavy, broader change sets, more review-loop churn
- Final story (5.9): test-only, most complex fixtures, significant review-loop on integration isolation
