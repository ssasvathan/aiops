## Key Insights and Lessons Learned

### L1: ATDD Assertions Must Cover Specification Completeness, Not Just Event Existence

Epic 4's code review found the same category of gap as Epics 2 and 3: a requirement was specified, the ATDD test existed, but the assertion didn't fully verify the requirement. The specific instance: suppression log events' ATDD test verified event name but not NFR-A3 required fields (`metric`, `cycle_timestamp`).

**Pattern:** "Does this event fire?" is a weaker assertion than "Does this event fire with all required fields?" For log event tests, assert the specific field values required by the NFR or FR, not just event presence.

**Apply going forward:** When writing ATDD tests for log events, derive assertions directly from the FR/NFR specification. NFR-A3: scope + metric + reason + cycle_timestamp. Test all four, not just event name.

---

### L2: Pre-Written TDD Helpers Are First-Class Code and Must Verify Call Signatures

Story 4-2's test infrastructure bugs were in pre-written TDD RED phase helpers — not in production code. Lambda signatures that use `**kwargs` but receive positional args, or helpers that forget to patch all external dependencies, cause silent failures that require debugging.

**Pattern:** Pre-written test helpers in the TDD RED phase should be reviewed with the same rigor as production code. Lambda signatures are particularly fragile. `*args, **kwargs` or explicit positional signatures should be used deliberately and verified.

**Apply going forward:** When creating TDD RED phase test helpers, include a comment documenting the expected call signature. Verify lambda signatures against the actual production function signatures before using the helper as a test foundation.

---

### L3: The Audit-First Pattern for Observability Stories Avoids Duplication

Story 4-2 began with a full audit of all 10 P6 log events before writing any code. This prevented duplicating work already done in Story 2.3. The "Critical Pre-Implementation Finding" section in the Dev Notes captured the audit result before implementation began.

**Pattern:** Observability instrumentation stories often land on partially-instrumented code because developers add logging/metrics during feature implementation. An audit-first pass prevents duplication and precisely scopes the remaining work.

**Apply going forward:** For stories that add observability to existing code (log events, OTLP, health registration), always audit current state before writing a line of code. Document the current state explicitly in Dev Notes as "already present" or "missing" before writing tasks.

---

### L4: The `_NamedWrapper` Pattern for SDK Objects That Lack Required Attributes

Story 4-1's `_NamedCounter`/`_NamedHistogram` wrappers solve a real category of problem: SDK objects that don't expose attributes needed for test assertion or runtime verification. The pattern is: wrap at creation, store the attribute locally, delegate all SDK methods.

**Pattern characteristics:**
- Wrap at the module boundary where instruments are created
- Store the "missing" attribute (`.name`) as a simple field
- Delegate `.add()` / `.record()` to the underlying SDK object unchanged
- Zero production behavior change; test assertions become straightforward

**Apply going forward:** When an SDK or framework object doesn't expose an attribute needed for testing or runtime checking, prefer a thin wrapper at the creation boundary over monkey-patching the SDK or adding runtime name-lookup hacks.
