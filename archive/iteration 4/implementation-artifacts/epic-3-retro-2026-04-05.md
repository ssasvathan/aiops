## What Went Well

### 1. Architectural Pre-Planning Eliminated the Biggest Risk

Winston (Architect): "The single most important thing in Epic 3 was the §Critical Architecture Gap section in Story 3-1's Dev Notes. We documented — before a single line of code was written — that `TriageExcerptV1.findings` uses `gate_input.Finding`, not `AnomalyFinding`, and therefore `baseline_context` numerical values are NOT accessible in the prompt layer."

Amelia (Dev): "That note was the difference between a clean first-pass and a potential dead-end. If I'd assumed `AnomalyFinding` was available in `TriageExcerptV1`, I'd have written code that imported `BaselineDeviationContext` and then discovered at runtime — or in a review — that it doesn't exist at that layer. The dev note pointed me directly to the `reason_codes` string-parsing approach."

Quinn (QA): "And the test fixtures in both stories confirm it — `_make_baseline_deviation_excerpt()` uses `contracts.gate_input.Finding`, not `models.anomaly.AnomalyFinding`. That architectural constraint was correctly propagated from Dev Notes into test helpers."

**Key win:** The `Finding` vs `AnomalyFinding` architectural constraint was fully documented before development began, resulting in a correct first-pass implementation.

**Lesson:** ATDD tests must be executed and green BEFORE requesting code review. A story where ATDD tests exist but have not been run against the implementation is not review-ready.

### 2. FR26 Numerical Values Not Available in Prompt — Architectural Constraint Limits Richness

Quinn (QA): "FR26 says 'include all deviating metrics with their values, deviation directions, magnitudes, baseline values, and current values.' The prompt actually includes metric names and directions, but NOT the numerical values — because `baseline_value`, `current_value`, and `deviation_magnitude` live on `AnomalyFinding.baseline_context`, which is never surfaced in `TriageExcerptV1`."

Winston (Architect): "This is a deliberate architectural boundary. The gate-layer `Finding` in `TriageExcerptV1` is a summary contract for routing and gating, not a full evidence record. The numerical values are intentionally not carried through."

Amelia (Dev): "The Story 3-1 Dev Notes document this explicitly: 'The architecture intentionally gate-layer `Finding` is a summary for routing/gating, not a full evidence record.' FR26 compliance is satisfied by the available data — metric names and directions from `reason_codes`."

Bob (Scrum Master): "But this is still a gap relative to what FR26 intended. The LLM gets less context than specified. That matters for hypothesis quality."

Winston (Architect): "The right fix, if we want full FR26 compliance, is an architectural evolution: either extend `TriageExcerptV1.findings` to carry `BaselineDeviationContext` numerical data, or pass the case file object itself to the prompt builder rather than the triage excerpt. Neither is an Epic 3 story — but it's a documented constraint."

**Lesson:** The `TriageExcerptV1` / `AnomalyFinding` architectural boundary limits LLM prompt richness. Full FR26 compliance requires either `TriageExcerptV1` enrichment or a direct case-file-to-prompt path. Document as a known architectural constraint for future consideration.

## Key Insights and Lessons Learned

### L1: ATDD Tests Must Be Executed Before Review Is Requested

Epic 3 repeated the Epic 2 pattern: a guard specified in the task checklist was marked done without the guard being in the code. Code review caught it both times. The common factor is that ATDD tests were written but not run against the implementation before review was requested.

**Apply to Epic 4:** Stories must include an explicit self-check: "Have I run the full ATDD test suite for this story against my implementation? Are all tests green?" This is not optional — it is the implementation-readiness gate before code review.

---

### L2: Verification-Only Stories Are a First-Class Delivery Pattern

Story 3-2 delivered real value with zero production code changes. The 5 new tests provide explicit test-evidence that the fallback path works for BASELINE_DEVIATION — a confidence guarantee that previously only existed by architectural reasoning. Test-evidenced architecture is materially more reliable than reasoning-only architecture.

**Apply broadly:** When the architecture already provides a behavior for a new case, a verification story that proves it with explicit tests is the right delivery vehicle — not an unnecessary story, not scope to skip.

---

### L3: The `TriageExcerptV1` Boundary Is a Known LLM Context Limiter

The gate-layer `Finding` in `TriageExcerptV1` intentionally does not carry `BaselineDeviationContext` numerical fields. This is correct architecture for gating/routing purposes. But it means the LLM prompt for BASELINE_DEVIATION cases lacks `baseline_value`, `current_value`, and `deviation_magnitude` — the most contextually rich fields for forming a hypothesis.

**Apply to future planning:** If LLM hypothesis quality for BASELINE_DEVIATION cases is ever evaluated and found insufficient, the root cause is this architectural boundary. The solution path is either: (A) enrich `TriageExcerptV1.findings` with a `BaselineDeviationContext` summary, or (B) pass the full case file object to the prompt builder. Document as a known, intentional constraint in `docs/architecture.md`.

---

### L4: Consistency Auditing Within Files Is a Review Discipline

Two findings in Epic 3 (M1 in 3-1 and M-2 in 3-2) were about internal file inconsistency: a function-body constant where module-level is the convention, and a missing gap assertion that all parallel tests have. Neither was caught by tests — both required a reviewer who read the full file.

**Apply to Epic 4:** Before submitting for review, do a consistency pass: for each new construct added to an existing file, check that it follows the same patterns as similar constructs already in that file.

---

### L5: Cold-Path LangGraph Isolation Is Proven and Stable

Epic 3 explicitly verified the D6 invariant through Story 3-2: `diagnosis/` is a cold-path leaf with no import path to the hot path. `build_fallback_report()` is a pure function. `build_llm_prompt()` is a pure function. The async cold-path diagnosis never blocks the hot path. This is not theoretical — it is now test-evidenced.

**Apply to Epic 4:** Epic 4 adds observability instrumentation. The OTLP counters and histograms must follow the same isolation discipline: no instrumentation on the hot path should depend on cold-path state. Health signals flow through `HealthRegistry` which is already thread/async-safe.

---

### L6: Two-Story Epics Can Deliver Full FR Coverage When Stories Are Well-Scoped

Epic 3 covered FR25-FR28 in 2 stories. Story 3-1 was substantive (3 source files modified, 4 new tests, 6 review findings). Story 3-2 was verification-only (0 source files modified, 5 new tests, 5 review findings). Together they provided: implementation + test evidence of the implementation + test evidence of the fallback path.

The asymmetry was by design — not all stories need to be the same size. Scope the story to what it actually needs to deliver, not to fill a sprint quota.

**Apply broadly:** Epic 4's 2 stories (4-1 and 4-2) should be similarly scoped to what they actually need. Don't artificially expand scope to fill story size expectations.
