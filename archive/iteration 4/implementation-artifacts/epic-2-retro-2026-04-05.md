## Challenges and Areas to Improve

### 1. `math.isfinite` Guard Was Specified in Task Checklist But Not Implemented

Bob (Scrum Master): "Story 2-1 had a Critical finding in code review: Task 1.2 was marked `[x]` for 'add `import math` (for `math.isfinite` guard)' — but the implementation never added it. NaN and Inf inputs silently produced `MADResult` objects with `is_deviation=False` instead of returning `None`."

Amelia (Dev): "That's an honest mistake. The task said to add the guard, I marked it done, but it never made it into the code. The code review caught it."

Quinn (QA): "And critically, the ATDD tests didn't cover NaN/Inf inputs — those test cases were only added in the code review phase. The pre-written tests focused on the happy paths and edge cases specified in the ACs, but not on adversarial input validation."

Winston (Architect): "The lesson here is: if a Task checklist item specifies adding a guard, there should be a test verifying that guard. The guard and its test should be linked at the story specification level."

**Lesson:** When a task checklist item specifies an input guard or validation, include an explicit ATDD test case for that guard in the pre-written test suite.

### 3. Logger Instantiation Pattern for structlog Test Isolation Was Non-Obvious

Amelia (Dev): "Story 2-3 had a real debugging session around the logger. The spec said 'add module-level logger.' But when you instantiate a structlog logger at module level, it captures the global structlog state at import time. The test `log_stream` fixture calls `structlog.reset_defaults()` after import, so a module-level logger never sees the JSON pipeline configuration."

Quinn (QA): "The fix was to instantiate the logger inside the function body. But this contradicts the 'module-level logger pattern' that's elsewhere in the codebase."

Winston (Architect): "This is a structlog-specific behavior that should be documented in `project-context.md`. When a function uses structlog and needs to be testable with fixture-level log reconfiguration, the logger must be instantiated lazily inside the function. This isn't true for all logging frameworks — it's structlog-specific."

Bob (Scrum Master): "This is the kind of thing that slows down Story 2-3 implementations if the next developer doesn't know about it. We should document it."

**Lesson:** Add a project-context rule: structlog loggers used in pipeline stage functions must be instantiated inside the function body (not at module level) to enable test fixture log reconfiguration.

---

### 4. `raise exc` vs. Bare `raise` — Python Traceback Semantics

Amelia (Dev): "Story 2-3's code review found `raise exc` in an except block — this replaces the original traceback with the re-raise point, losing the original exception context. The fix is a bare `raise`, which re-raises preserving the original traceback."

Quinn (QA): "This is a subtle Python gotcha. `raise exc` looks correct but isn't. The difference matters in production: when you see the traceback, you want to know where the original exception occurred, not where it was re-raised."

Winston (Architect): "This should go into project-context anti-patterns. 'Never use `raise exc` inside an except block that caught `exc`. Always use bare `raise`.'"

**Lesson:** Add to project anti-patterns: `raise exc` inside an `except exc` block. Use bare `raise` to preserve traceback chain.

---

### 5. Suppression Log Level (INFO vs DEBUG) Requires AC Clarity

Amelia (Dev): "Story 2-3 logged `baseline_deviation_suppressed_single_metric` at INFO, but the code review found that AC3 specifies DEBUG. The issue was ambiguous language in the acceptance criteria — 'logging them at DEBUG level for diagnostic purposes' in the FR specification, but the AC3 output clause didn't repeat the log level."

Quinn (QA): "The consequence was that the `test_single_metric_suppressed_log_event` test used the shared `log_stream` fixture (INFO level) instead of a `debug_log_stream` fixture. When the level was corrected to DEBUG, the test had to be updated to use a DEBUG-level fixture — which then had to be created as a file-local fixture."

Bob (Scrum Master): "The fix: ACs that specify structured log events should always include the log level explicitly. 'A `baseline_deviation_suppressed_single_metric` DEBUG log event is emitted with scope, metric_key, and reason code.'"

**Lesson:** Structured log event acceptance criteria must explicitly state the log level. Ambiguous log level → wrong implementation → test rewrite at review time.

## Key Insights and Lessons Learned

### L1: Universal ATDD Is the Right Default — But Test Specs Need Input Guard Coverage

Epic 2 applied ATDD universally across all 4 stories and it consistently produced cleaner, faster implementations than the ad-hoc approach. However, Story 2-1 revealed a gap: pre-written ATDD tests naturally focus on functional behavior specified in ACs, but they often miss adversarial input scenarios (NaN, Inf, None). These only surfaced in code review.

**Apply to Epic 3:** When a story's Dev Notes specify input guards or validation paths, include explicit ATDD tests for those guards — not just the happy-path ACs. Mark them in the spec as "guard test: verify guard_condition returns None/raises."

---

### L2: Cross-Module Type Dependencies Need Explicit Architecture Pre-Planning

Story 2-2's circular import between `baseline/models.py` and `models/anomaly.py` was solvable but required non-trivial Pydantic v2 knowledge (`from __future__ import annotations` + `TYPE_CHECKING` + `model_rebuild()`). It cost debugging time.

**Apply to Epic 3:** Epic 3 involves `diagnosis/` cold-path code consuming `AnomalyFinding` and `BaselineDeviationContext`. Review cross-module import directions before Story 3-1 is created. If new circular import risks are identified, document the resolution pattern in the story Dev Notes.

---

### L3: structlog Logger Instantiation Is Test-Context-Sensitive

Module-level logger instantiation in pipeline stage functions breaks structlog test fixture reconfiguration. This is a structlog-specific behavior that conflicts with the general "module-level imports only" rule applied to test files. The pattern is: lazy logger instantiation inside the function body for pipeline stages that need to be tested with log stream fixtures.

**Apply to Epic 3:** Add this to `project-context.md` as a framework-specific rule. Story 3-1 (LLM diagnosis cold-path) will have structured log events — this pattern applies directly.

---

### L4: Python `raise exc` vs. Bare `raise` Should Be an Anti-Pattern Rule

This is subtle enough that it appeared in review without any prior project documentation. It affects traceability in production debugging — preserving the original traceback is high-value, zero-effort with a bare `raise`.

**Apply to Epic 3:** Add to `project-context.md` anti-patterns section before Epic 3 begins.

---

### L5: The Additive-Only Schema Evolution Pattern (Procedure A) Scales Well

Epic 2 added `BASELINE_DEVIATION` to three separate Literals: `AnomalyFamily` in `models/anomaly.py`, `AnomalyFamily` in `contracts/gate_input.py`, and throughout `gating.py` annotations. Each extension was additive-only with a default field. Zero regressions across 1,309 tests confirm the pattern is robust.

**Apply to Epic 3:** Epic 3 will likely not extend any existing Literals but will add new models to `diagnosis/`. Follow the same frozen Pydantic model pattern (P4/P5) established in Epic 2.

---

### L6: The Read-Then-Write Ordering for Baseline Buckets Is Critical and Easy to Violate

Story 2-4's `_update_baseline_buckets()` is called AFTER `collect_baseline_deviation_stage_output()` returns. This ordering ensures detection reads the pre-cycle historical values and writes the current observation afterward. Getting this wrong (writing before reading) would contaminate the detection baseline with the current observation — a subtle data correctness bug.

**Apply to Epic 4:** When the OTLP observability story adds latency metrics, ensure the `update_bucket()` timing is measured as its own distinct phase (post-detection) so the metric doesn't misleadingly appear to be part of detection latency.

---

### L7: `_AnomalyFamily` Type Alias Is a Maintainability Win for Future Family Extensions

Epic 3 and 4 likely won't add new anomaly families, but if a future feature does, the `_AnomalyFamily` alias in `gating.py` means exactly one change instead of 5+. This was an unprompted improvement from the Story 2-4 implementation — worth calling out as a positive pattern.

**Apply broadly:** When a Literal type is used in 3+ locations in the same file, extract it as a module-private alias.
