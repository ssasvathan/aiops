## What Went Well

1. **Well-planned story sequencing** — each story consumed what the previous one built; no rework of foundations mid-epic
2. **Consistent architectural patterns** — `frozen=True` Pydantic models, `@functools.cache` asyncio-safe singletons, structlog processor pipeline all established early and used consistently through all 9 stories
3. **Code review as a quality engine** — caught real bugs (Kafka listener, OTLP UpDownCounter delta API, contextvars surgical unbind) and delivered an unrequested improvement (multi-stage Dockerfile, -200MB runtime image)
4. **Test suite growth with increasing rigor** — 149 tests with coverage expanding per story
5. **Clean architectural boundaries** — `harness/` separated from `src/`, asyncio conventions consistent, structlog pipeline extensible

## Challenges

1. **Test edge-case coverage gaps in 9/9 stories** — missing tests (error branches, boundary conditions, edge cases) surfaced at review rather than during dev in every story
2. **Docker quality gates deferred** — AI dev environment has no Docker daemon; smoke-test and container integration unexecuted until project lead ran manually (now closed)
3. **Configuration and API contract bugs caught late** — Kafka `ADVERTISED_LISTENERS`, OTEL `UpDownCounter` delta vs absolute, `clear_contextvars` vs `unbind_contextvars` — each was an implementation assumption caught at review, not during dev
4. **Story 1.6 required three review rounds** — asyncio concurrency surface (health server try/finally, UpDownCounter API, AwareDatetime) required multiple passes

## Key Insights

1. **AC edge-case specificity is the upstream fix** — if the acceptance criteria explicitly name error branches and boundary conditions, the definition of done becomes self-enforcing and review becomes a verification step rather than a discovery step
2. **Review process is functioning well** — the issue is pre-review, not the review itself
3. **Docker-less AI dev environments create a structural gap** for infrastructure-heavy stories — must be planned for explicitly in AC
4. **The harness traffic patterns are the test oracle for Epic 2** — `consumer_lag`, `throughput_proxy`, `volume_drop`, `normal` patterns confirmed working in live stack; anomaly detection correctness in Epic 2 will be validated against these signals
